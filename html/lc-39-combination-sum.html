Combination Sum

Find all combinations that sum to a target — backtracking with pruning and unlimited reuse

MEDIUM
LC 39

Problem Statement

Challenge

Given a list of distinct integers nums and an integer target, return all unique combinations where chosen numbers sum to target.

  • You can use each value in nums unlimited times.
  • Two combinations are the same if they contain the same values with the same frequencies.
  • Output order does not matter.

Example 1

Input / Output
Input : nums = [2, 5, 6, 9], target = 9 Output: [[2, 2, 5], [9]] # 2 + 2 + 5 = 9 and 9 = 9.

Example 2

Input / Output
Input : nums = [3, 4, 5], target = 16 Output: [[3,3,3,3,4], [3,3,5,5], [3,4,4,5], [4,4,4,4]] # All unique frequency combos that sum to 16.
💡
Key Insight: Build combinations in non-decreasing index order so [2,5,2] is never generated separately from [2,2,5]. Reusing a number is achieved by recursing with the same index j. Sorting enables early pruning: once total + nums[j] > target, all later values are also too large.

Core Concepts

Pattern Definition

State is a partially built combination cur and current sum total. At each state, choose next candidate from current start index onward. Recurse with the same index to allow unlimited reuse, then backtrack.

Core Loop

Python — Pattern Skeleton
def dfs(i, total): if total == target: record(cur.copy()) return for j in range(i, len(nums)): if total + nums[j] > target: break cur.append(nums[j]) dfs(j, total + nums[j]) cur.pop()
⚠️
Important Caveat: After choosing nums[j], recurse with dfs(j, ...) — not dfs(j+1, ...). Using j+1 forbids reuse and misses combinations like [2,2,3].

Decision Process / State View

State at each recursive call: (i, cur, total)

  • i — minimum index allowed next (keeps combinations unique)
  • cur — current chosen numbers
  • total — sum of numbers in cur
Decision Tree — nums = [2, 5, 6, 9], target = 9
dfs(i=0, cur=[], total=0) ├── choose 2 → dfs(i=0, cur=[2], total=2) │ ├── choose 2 → dfs(i=0, cur=[2,2], total=4) │ │ ├── choose 2 → dfs(i=0, cur=[2,2,2], total=6) │ │ │ ├── choose 2 → total=8 → next 2 would be 10 → prune │ │ │ └── choose 5 → total=9 → record [2,2,5] ✓ │ │ └── choose 5 → total=9 → record [2,2,5] already found at lower branch │ └── (deeper branches all exceed or miss target) └── choose 9 → dfs(i=3, cur=[9], total=9) → record [9] ✓
📊
Max recursion depth ≈ target / min(nums). For target=9, min=2: depth ≈ 4.

Step-by-Step Walkthrough

1

Initialize result, path, and sort

Python
res = [] cur = [] nums.sort()

res stores complete answers, cur is the current path; sorting enables early pruning.

2

Define DFS state dfs(i, total)

Python
def dfs(i, total): ...

i controls allowed start index (uniqueness); total tracks current sum.

3

Base case: target reached → record combination

Python
if total == target: res.append(cur.copy()) return

This path is a valid combination. Copy is essential — cur is mutated by backtracking.

4

Iterate candidates from index i onward

Python
for j in range(i, len(nums)): ...

Starting from i (not 0) prevents generating permutations of the same combination.

5

Prune impossible branches

Python
if total + nums[j] > target: break

Sorted array guarantees all later numbers are also too large — break cuts the whole suffix.

6

Choose current number and recurse

Python
cur.append(nums[j]) dfs(j, total + nums[j])

Pass j (not j+1) so the same number can be picked again — unlimited reuse.

7

Backtrack to restore state

Python
cur.pop()

Removes last decision before trying next candidate. State is restored for sibling branches.

8

Start DFS and return result

Python
dfs(0, 0) return res

Begin with empty path and sum zero; return all found combinations.

Complete Solution (Primary)

Python
from typing import List class Solution: def combinationSum(self, nums: List[int], target: int) -> List[List[int]]: nums.sort() res = [] cur = [] def dfs(i, total): if total == target: res.append(cur.copy()) return for j in range(i, len(nums)): nxt = total + nums[j] if nxt > target: break cur.append(nums[j]) dfs(j, nxt) cur.pop() dfs(0, 0) return res print(Solution().combinationSum([2, 5, 6, 9], 9)) # Expected: [[2, 2, 5], [9]] (order may vary)

Step Events

  • enter — New DFS call: i and total set; cur unchanged.
  • choose — Pick nums[j]: append to cur; total becomes total + nums[j].
  • record — Base case: copy of cur appended to res; res_count += 1.
  • prune — total + nums[j] > target: skip this branch and all right siblings.
  • backtrack — After recursive call: cur.pop(); total restored.

Knowledge Check

Q1 — Complexity: What is the typical backtracking time complexity for Combination Sum?

A) O(n)
B) O(n log n)
C) O(2^(t/m)) where t = target, m = min(nums)
D) O(t²)
Answer: C — Recursion depth is bounded by target / min(nums). At each level there are multiple branching choices, giving an exponential tree of size O(2^(t/m)).

Q2 — Correctness Invariant: Which invariant prevents duplicate permutations like [2,5,2] and [2,2,5] both appearing?

A) We sort the final result and deduplicate.
B) We always recurse with j + 1.
C) We only choose indices from the current i forward, keeping each path in non-decreasing index order.
D) We use a set of tuples to de-duplicate every path.
Answer: C — Restricting candidates to range(i, len(nums)) means once we move past an index, we never revisit it, so every combination is generated in canonical non-decreasing order.

Q3 — Edge Case: What does the algorithm return for nums = [3], target = 5?

A) [[3, 2]]
B) [[3]]
C) []
D) Raises a RuntimeError
Answer: C — 3 cannot be repeatedly added to reach exactly 5 (3, 6, 9… — none equals 5). The pruning fires at 6 and no combination is recorded, so the result is empty [].

Q4 — Implementation Pitfall: After choosing nums[j], the next recursive call should be…

A) dfs(j + 1, total + nums[j]) — advance to next index
B) dfs(0, total + nums[j]) — restart from beginning
C) dfs(j, total + nums[j]) — stay at same index for reuse
D) dfs(i + 1, total + nums[j]) — advance outer loop index
Answer: C — Passing j (not j+1) allows the same number to be picked again, which is required since each number can be used unlimited times. Using j+1 would miss combinations like [2,2,3].

Alternative Approaches

Alternative 1: Include/Skip Binary Backtracking

Intuition: At each index, either include the current value (stay at same index for reuse) or skip to the next index. Two branches per call — matches the classic subset decision tree.

Python
from typing import List def combinationSum_include_skip(nums: List[int], target: int) -> List[List[int]]: res = [] cur = [] def dfs(i, total): if total == target: res.append(cur.copy()) return if i >= len(nums) or total > target: return cur.append(nums[i]) dfs(i, total + nums[i]) # include: stay at i cur.pop() dfs(i + 1, total) # skip: move to i+1 dfs(0, 0) return res print(combinationSum_include_skip([2, 5, 6, 9], 9)) # Expected (order may vary): [[2, 2, 5], [9]]
  • Time: O(2^(t/m))
  • Space: O(t/m) recursion depth.
  • When to prefer: Teaching the classic include/exclude decision-tree pattern; makes the binary branching structure explicit.

Alternative 2: DP by Sum (Build Combinations)

Intuition: Build combinations for each sum from 0 to target. dp[s] holds all combinations summing to s. Extend each using new candidates in non-decreasing order to avoid duplicates.

Python
from typing import List def combinationSum_dp(nums: List[int], target: int) -> List[List[int]]: nums.sort() dp = [[] for _ in range(target + 1)] dp[0] = [[]] for num in nums: for s in range(num, target + 1): for prev in dp[s - num]: if not prev or prev[-1] <= num: dp[s].append(prev + [num]) return dp[target] print(combinationSum_dp([2, 5, 6, 9], 9)) # Expected (order may vary): [[2, 2, 5], [9]]
  • Time: roughly O(target × K) where K depends on combination counts.
  • Space: O(total combinations stored across dp table).
  • When to prefer: When you want bottom-up construction with no recursion stack.

Common Mistakes

Mistake 1: Recursing with j+1 — breaks unlimited reuse

Wrong
cur.append(nums[j]) dfs(j + 1, total + nums[j]) # BUG cur.pop() # misses [2,2,3] for target 7 on [2,3,6,7]
Correct
cur.append(nums[j]) dfs(j, total + nums[j]) # stay at j cur.pop() # correctly includes [2,2,3]

Why it fails: Advancing to j+1 forbids picking the same number again, missing all combinations that reuse values.

Quick check: Verify [2,2,3] appears in the output for nums=[2,3,6,7], target=7.

Mistake 2: Forgetting to backtrack — corrupted paths

Wrong
for j in range(i, len(nums)): if total + nums[j] > target: break cur.append(nums[j]) dfs(j, total + nums[j]) # missing cur.pop() — BUG
Correct
for j in range(i, len(nums)): if total + nums[j] > target: break cur.append(nums[j]) dfs(j, total + nums[j]) cur.pop() # restore state

Why it fails: Without cur.pop(), sibling branches inherit stale choices from prior iterations, producing corrupted paths.

Quick check: Ensure cur length returns to its previous size after each recursive call.

Mistake 3: Missing pruning — explores dead branches

Wrong (inefficient)
for j in range(i, len(nums)): cur.append(nums[j]) dfs(j, total + nums[j]) cur.pop() # no pruning — explores all branches
Correct
for j in range(i, len(nums)): if total + nums[j] > target: break # pruning cur.append(nums[j]) dfs(j, total + nums[j]) cur.pop()

Why it fails: Without break, the algorithm explores branches that can never reach the target, wasting work on a sorted array.

Quick check: With sorted nums, always use if total + nums[j] > target: break.

Mistake 4: Appending cur reference instead of a copy

Wrong
if total == target: res.append(cur) # stores reference return # all entries in res will be mutated # by subsequent pop() calls
Correct
if total == target: res.append(cur.copy()) # snapshot return # each entry is an independent list

Why it fails: cur is mutated by every pop() during backtracking. All entries in res that point to the same object get corrupted.

Quick check: Always save cur.copy(); verify each element of res is a distinct object with id().

When do we record and stop a path in Combination Sum?

(click to flip)

Record when total == target — copy cur and return. Prune a branch when the next sorted candidate would push total over target (break out of the loop).

Why is the time complexity exponential, and what controls the exponent?

(click to flip)

The recursion depth is bounded by target / min(nums) (max reuses of the smallest value). Each level branches over multiple candidates, giving O(2^(t/m)) total states.

What ensures uniqueness — no duplicate permutations — without a set?

(click to flip)

Restrict choices to indices j ≥ i and recurse with the same j for reuse. This keeps each path in non-decreasing index order — every combination is generated exactly once in canonical form.
